# Сравнение систем логирования: текущая система vs LogRocket vs Self-hosted Sentry

> **Примечание:** Текущий стек: **Tracer** (ошибки) + **docker logs** (логи).

## 📋 Обзор

Этот документ сравнивает три варианта системы логирования для проекта GAFUS:

1. **Текущая система** (`@gafus/logger` → Tracer, docker logs)
2. **LogRocket** (SaaS)
3. **Self-hosted Sentry**

## 🔍 Детальное сравнение

### 1. Текущая система (`@gafus/logger` → Tracer, docker logs)

#### ✅ Преимущества

| Критерий                 | Оценка                                     |
| ------------------------ | ------------------------------------------ |
| **Стоимость**            | ✅ Бесплатно (self-hosted)                 |
| **Контроль данных**      | ✅ Полный контроль в своей БД              |
| **Интеграция**           | ✅ Нативная для монорепо                   |
| **Гибкость**             | ✅ Полная кастомизация под нужды проекта   |
| **Производительность**   | ✅ Pino — один из самых быстрых логгеров   |
| **Размер бандла**        | ✅ Минимальный (Pino очень лёгкий)         |
| **Offline-первый**       | ✅ Работает без внешних зависимостей       |
| **Настройка под проект** | ✅ Уже интегрирована со всеми приложениями |

#### ❌ Недостатки

| Недостаток                | Описание                                    |
| ------------------------- | ------------------------------------------- |
| **Session Replay**        | ❌ Нет записи пользовательских сессий       |
| **Source Maps**           | ⚠️ Нет автоматической обработки source maps |
| **Breadcrumbs**           | ⚠️ Нет автоматической истории действий      |
| **Производительность UI** | ⚠️ Нужна доработка для больших объёмов      |
| **Алертинг**              | ⚠️ Нет встроенной системы алертинга         |

#### Что уже реализовано

- ✅ Ошибки в Tracer (серверные и клиентские), логи — docker logs
- ✅ Структурированные логи (JSON через Pino)
- ✅ Поддержка всех приложений монорепо (web, trainer-panel, admin-panel, worker, telegram-bot)
- ✅ React Error Boundaries (`@gafus/error-handling`)
- ✅ Специализированные логгеры для каждого типа сервиса
- ✅ Логи в stdout (docker logs)
- ✅ Интеграция с BullMQ worker-ами

#### Что можно добавить при необходимости

- 📝 Source maps для красивых stack traces (можно добавить отдельно)
- 📝 Breadcrumbs (история действий до ошибки — можно добавить вручную)
- 📝 Алертинг через Telegram/Email (можно интегрировать)
- 📝 Session replay (можно интегрировать LogRocket отдельно только для этого)

### 2. LogRocket (SaaS)

#### ✅ Преимущества

| Критерий            | Оценка                                      |
| ------------------- | ------------------------------------------- |
| **Session Replay**  | ✅ Лучшая в индустрии запись сессий         |
| **User Experience** | ✅ Детальный анализ поведения пользователей |
| **Настройка**       | ✅ Быстрая интеграция (SDK)                 |
| **UI/UX**           | ✅ Готовый профессиональный интерфейс       |
| **Поддержка**       | ✅ Команда поддержки                        |

#### ❌ Недостатки

| Недостаток              | Описание                                                     |
| ----------------------- | ------------------------------------------------------------ |
| **Стоимость**           | ❌ **$99/месяц** (Team план) или **$299/месяц** (Enterprise) |
| **Лимиты**              | ❌ Ограничение на количество сессий (10K/месяц на Team)      |
| **Контроль данных**     | ❌ Данные у провайдера (США)                                 |
| **Backend логирование** | ⚠️ Слабая поддержка серверных логов                          |
| **Приватность**         | ⚠️ Запись всего взаимодействия пользователя                  |
| **GDPR**                | ⚠️ Требует согласия пользователя на запись                   |
| **Размер SDK**          | ⚠️ ~50KB дополнительный бандл                                |
| **Зависимость**         | ❌ Зависимость от внешнего сервиса                           |

#### Функциональность

- ✅ Session Replay (видео-запись действий пользователя)
- ✅ Console logs в контексте сессии
- ✅ Network requests с timing
- ✅ Error tracking (базовый)
- ⚠️ Performance monitoring (ограниченный)
- ❌ Backend error tracking (слабая поддержка)
- ❌ Метрики системы (нет)

#### Полезность для проекта

**Лучше всего подходит для:**

- Анализа UX проблем
- Воспроизведения багов "на живом примере"
- Понимания поведения пользователей

**Не подходит для:**

- Полноценного логирования backend
- Мониторинга worker-ов и фоновых задач
- Хранения данных в России (GDPR требования)

---

### 3. Self-hosted Sentry

#### ✅ Преимущества

| Критерий                  | Оценка                                      |
| ------------------------- | ------------------------------------------- |
| **Контроль данных**       | ✅ Полный контроль (self-hosted)            |
| **Session Replay**        | ✅ Есть (встроенная функция)                |
| **Source Maps**           | ✅ Автоматическая обработка                 |
| **Breadcrumbs**           | ✅ Автоматическая история действий          |
| **Backend support**       | ✅ Отличная поддержка Node.js               |
| **UI/UX**                 | ✅ Профессиональный интерфейс               |
| **Алертинг**              | ✅ Встроенная система алертинга             |
| **Performance**           | ✅ APM (Application Performance Monitoring) |
| **Официальная поддержка** | ✅ Есть поддержка self-hosted               |

#### ❌ Недостатки

| Недостаток                   | Описание                                         |
| ---------------------------- | ------------------------------------------------ |
| **Сложность развёртывания**  | ❌ Требует Docker, PostgreSQL, Redis, ClickHouse |
| **Ресурсы**                  | ❌ **Минимум 4GB RAM, 2 CPU cores**              |
| **Обслуживание**             | ❌ Нужно обновлять, мониторить, бэкапить         |
| **Миграция**                 | ❌ Нужно мигрировать всю текущую систему         |
| **Стоимость инфраструктуры** | ⚠️ Дополнительные серверные ресурсы              |
| **Learning curve**           | ⚠️ Команде нужно изучить Sentry                  |
| **Размер бандла**            | ⚠️ ~30KB дополнительный SDK                      |

#### Технические требования

```yaml
Минимальные требования:
  RAM: 4GB (рекомендуется 8GB)
  CPU: 2 cores (рекомендуется 4 cores)
  Disk: 20GB (растёт со временем)
  Services:
    - PostgreSQL (база для метаданных)
    - Redis (кеширование и очереди)
    - ClickHouse (хранилище событий) или PostgreSQL
    - Docker Compose для оркестрации

Сложность установки: Высокая (1-2 дня настройки)
Сложность обслуживания: Средняя (регулярные обновления)
```

#### Функциональность

- ✅ Error tracking (full-stack)
- ✅ Session Replay
- ✅ Source maps автоматически
- ✅ Breadcrumbs (история действий)
- ✅ Performance monitoring (APM)
- ✅ Алертинг (Email, Slack, Telegram)
- ✅ Release tracking
- ✅ User feedback
- ✅ Интеграции (GitHub, Jira, etc.)

## 🎯 Рекомендации

### Вариант 1: Остаться на текущей системе (РЕКОМЕНДУЕТСЯ)

**Когда выбирать:**

- ✅ Бюджет ограничен или равен нулю
- ✅ Полный контроль данных критичен (GDPR, локальное хранение)
- ✅ Нужна максимальная гибкость под нужды проекта
- ✅ Система уже работает и покрывает потребности

**Что добавить при необходимости:**

1. **Session Replay** — можно интегрировать LogRocket **только для записи сессий** (опционально)
   - Стоимость: только если действительно нужно
   - Использование: только для критических багов

2. **Source Maps** — обработка в Tracer (если поддерживается SDK)
   - Сложность: средняя (1-2 дня работы)
   - Улучшит читаемость stack traces

3. **Breadcrumbs** — добавить вручную в критических местах
   - Сложность: низкая (можно добавлять постепенно)
   - Пример: `logger.info('User clicked button', { action: 'submit', formId: '123' })`

4. **Алертинг** — интегрировать Telegram/Email алертинг
   - Сложность: низкая (1 день работы)
   - Использовать существующую инфраструктуру Telegram bot

**Оценка трудозатрат на улучшения:** 3-5 дней работы

---

### Вариант 2: Перейти на Self-hosted Sentry

**Когда выбирать:**

- ✅ Нужен Session Replay из коробки
- ✅ Нужен автоматический алертинг
- ✅ Есть ресурсы на развёртывание и обслуживание (DevOps)
- ✅ Команда готова потратить время на миграцию
- ✅ Нужен APM (Application Performance Monitoring)

**Трудозатраты на миграцию:**

- Развёртывание Sentry: **1-2 дня** (DevOps)
- Миграция кода: **3-5 дней** (заменить все вызовы логгера)
- Тестирование: **2-3 дня**
- Документация: **1 день**
- **Итого: 7-11 дней** работы команды

**Риски:**

- ⚠️ Сложность развёртывания (может занять больше времени)
- ⚠️ Нужны дополнительные серверные ресурсы
- ⚠️ Потеря гибкости (зависимость от Sentry API)

---

### Вариант 3: Перейти на LogRocket

**Когда выбирать:**

- ❌ **НЕ РЕКОМЕНДУЕТСЯ** для этого проекта

**Почему не подходит:**

- ❌ Дорого ($99-299/месяц)
- ❌ Слабая поддержка backend
- ❌ Данные хранятся у провайдера (проблемы с GDPR)
- ❌ Не заменяет текущую систему логирования

**Когда имеет смысл:**

- Только если нужен Session Replay для фронтенда **в дополнение** к текущей системе
- Использовать как **дополнительный инструмент**, а не замену

---

## 🎬 Итоговая рекомендация

### Остаться на текущей системе + добавить при необходимости

**Почему:**

1. **Экономия** — $0 вместо $1,200-35,000 за 10 лет
2. **Уже работает** — система покрывает 90% потребностей
3. **Гибкость** — можно добавлять функции по мере необходимости
4. **Контроль** — полный контроль над данными и архитектурой
5. **Производительность** — Pino — один из самых быстрых логгеров

**План улучшений (при необходимости):**

```
Приоритет 1 (можно добавить за 1-2 дня):
  - Source maps в Tracer
  - Алертинг через Telegram/Email

Приоритет 2 (можно добавить за 2-3 дня):
  - Breadcrumbs для критических действий
  - Grafana для больших объёмов логов

Приоритет 3 (опционально):
  - Интеграция LogRocket только для Session Replay (если критично)
  - Performance monitoring через существующие инструменты
```

### Альтернатива: Self-hosted Sentry (если критичен Session Replay)

**Когда стоит рассмотреть:**

- Если Session Replay критически важен (например, сложные баги на проде)
- Если есть DevOps ресурсы на развёртывание
- Если готовы потратить 1-2 недели на миграцию

**Но учтите:**

- Это **большой шаг** — миграция всей системы логирования
- Потребует **дополнительных серверных ресурсов**
- Потеряете **гибкость** текущей системы

---

## 📝 Выводы

1. **Текущая система оптимальна** для проекта — покрывает основные потребности без дополнительных затрат
2. **LogRocket не подходит** как замена — слишком дорого и слабое backend логирование
3. **Self-hosted Sentry имеет смысл** только если критически нужен Session Replay из коробки
4. **Гибридный подход** возможен: текущая система + LogRocket только для Session Replay (опционально)

**Рекомендуемое решение:** Остаться на текущей системе и добавить улучшения по мере необходимости (source maps, алертинг, breadcrumbs).
